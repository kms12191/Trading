# ML 실전 실행 체크리스트

이 문서는 관리자 페이지 `/admin/ml-data`를 기준으로, 실제 데이터셋 수집부터 학습, 비교, 서비스 반영까지 어떤 순서로 진행하면 되는지 팀 공용 절차로 정리한 문서입니다.

---

## 1. 시작 전 확인

먼저 관리자 페이지의 **운영 준비 상태** 패널을 확인합니다.

반드시 우선 확인할 항목:

1. `Toss 키`가 `READY`인지
2. `주식 원천 CSV`, `코인 원천 CSV`가 이미 있는지
3. `매크로 지표`가 필요한 경우 준비됐는지
4. `외부 피처`가 비어 있는지, 일부라도 채워졌는지
5. 현재 `SERVING` 모델이 무엇인지

---

## 2. 키 저장 및 연결 확인

### 2.1 Toss

다음이 먼저 완료되어야 합니다.

* `API Key`
* `Secret Key`
* 필요 시 `accountSeq`

이 값은 프론트에 직접 넘기지 않고, Supabase `user_api_keys`에 암호화 저장된 뒤 백엔드에서만 복호화해 사용합니다.

### 2.2 Binance

현재 학습용 코인 캔들은 공개 API 기반이라 개인 키 없이도 수집 가능합니다.

---

## 3. 데이터셋 수집

### 3.1 주식

권장 버튼:

* `주식 v7 자동 수집+학습`

내부 기준:

* preset: `stock_core_90`
* interval: `1d`
* count: `700`
* macro 포함

### 3.2 코인

권장 버튼:

* `코인 v7 자동 수집+학습`

내부 기준:

* preset: `crypto_core_30`
* interval: `1h`
* count: `2500`

### 3.3 수동 수집이 필요한 경우

먼저 `CSV 생성`만 수행한 뒤, 실패 심볼과 행 수를 확인합니다.

확인 포인트:

* `failure_count`
* 실제 생성 행 수
* `ml/data/raw/*.csv` 갱신 여부

---

## 4. 학습 실행

자동 버튼을 사용하지 않는 경우 아래 중 하나를 실행합니다.

* `주식 v6 학습`
* `주식 v7 학습`
* `코인 v6 학습`
* `코인 v7 학습`

확인 포인트:

* 작업 이력 `success`
* summary JSON 생성 여부
* 모델 결과 카드 갱신 여부
* 최신 Markdown 실험 리포트 자동 생성 여부

---

## 5. 성능 비교

관리자 페이지에서 아래를 우선 비교합니다.

1. `SERVING`
2. `PICK`
3. `LATEST`
4. 버전 비교 표의 `CV 구분력`
5. `상위 10% 적중`
6. `복합 초과수익(순)`
7. `복합 승률`
8. `최대낙폭`

권장 해석:

* 주식은 `복합 초과수익(순)`이 플러스인지와 `CV 구분력`이 유지되는지 같이 봅니다.
* 코인은 `ROC AUC`만 좋고 비용 반영 수익이 음수일 수 있으므로 바로 serving으로 올리지 않습니다.

---

## 6. 서비스 반영

레지스트리 패널에서 진행합니다.

순서:

1. 검토할 버전의 `PICK`, `LATEST`, `SERVING` 상태 확인
2. 새 버전을 쓰기로 결정했으면 `서비스 반영` 버튼 클릭
3. `SERVING` 배지가 바뀌었는지 확인
4. 모델 결과 카드의 기본 선택 버전이 바뀌었는지 확인
5. `GET /api/ml/active-model?asset_type=STOCK|CRYPTO` 기준 선택 결과가 맞는지 확인
6. 최근 리포트 목록에 이번 실행 결과가 남았는지 확인

---

## 7. 실패 시 확인 순서

### 7.1 Toss 수집 실패

우선 확인:

1. 키 저장 여부
2. rate limit
3. 장외 시간 / 빈 캔들
4. 실패 심볼 목록

### 7.2 학습 실패

우선 확인:

1. 작업 이력의 stderr
2. 원천 CSV 행 수
3. 외부 피처 파일 유무
4. summary 파일 생성 여부

### 7.3 서비스 반영 실패

우선 확인:

1. 레지스트리 항목 존재 여부
2. Supabase 테이블 권한
3. 파일 레지스트리 fallback 동작 여부

---

## 8. 최종 결정 원칙

운영 반영은 단순히 최신 버전이라고 올리지 않습니다.

최종 기준:

1. `SERVING`보다 성능 개선이 실제로 있는지
2. `PICK`이 비용 반영 기준에서도 의미가 있는지
3. 특정 섹터/시장에만 과적합되지 않았는지
4. 재실행 시 결과가 너무 흔들리지 않는지

가능하면 최소 2회 이상 같은 조건 재검증 후 serving 전환을 권장합니다.
