# AI 위탁운용 1차 안정화 스펙

- 작성일: 2026-07-22
- 상태: 사용자 검토 대기
- 범위: 관리자용 AI 위탁운용을 단일 거래소 구현에서 거래소 공통 안전 운용 기반으로 확장
- 선행 문서: `docs/superpowers/specs/2026-07-22-admin-ai-fund-design.md`

## 1. 목표

1차는 수익 전략을 많이 늘리는 단계가 아니다. 코인원 중심으로 구현된 AI 위탁운용이 여러 거래소에서 일관된 규칙으로 주문되고, 장애·중복 실행·부분 체결 상황에서도 자금과 포지션을 정확하게 보존하도록 만드는 단계다.

완료 기준은 다음과 같다.

- 거래소별 API 차이를 공통 어댑터 계약으로 흡수한다.
- AI 위탁운용이 생성한 포지션과 수동 거래 포지션을 논리적으로 분리한다.
- 위탁운용 매도는 위탁 원장에 남은 수량을 초과하지 않는다.
- 주문은 생성부터 체결·취소·실패·검토 대기까지 추적 가능하다.
- 워커 재시작과 거래소 통신 실패 뒤에도 거래소 실상태와 내부 상태를 조정한다.
- 실거래는 명시적으로 활성화된 거래소와 유효한 사용자 API 키에서만 수행한다.

## 2. 범위와 제외 범위

### 포함

- 코인원, 토스증권, 바이낸스를 대상으로 한 공통 거래소 어댑터 계약
- 거래소별 거래 가능 여부와 주문 제약을 나타내는 capability 모델
- AI 위탁운용 전용 원장과 주문 원장
- 주문 생명주기 상태 머신 및 멱등성 키
- 잔고·주문·체결 조회 기반의 정합성 조정 작업
- 일간 손실 한도, 배정 자금 한도, 종목별 최대 비중, 중복 매수 차단
- 비정상 시 신규 진입 중단, 미확정 주문 검토 대기, 운영 알림용 이벤트 기록
- PAPER, CANARY, LIVE 운용 모드

### 제외

- DCA, 그리드, 트레일링 익절, 복수 목표가 분할 매도 같은 전략 고도화
- 사용자 공개형 자동투자 상품, 수수료·성과보수·계약 관리
- 관리자 권한 검증 체계의 전면 개편
- 고급 운영 대시보드와 외부 메신저 알림 연동

관리자 권한 검증은 현재 요청에 따라 후속 범위로 둔다. 다만 모든 새 API와 워커 경로는 기존 관리자 경계를 우회하지 않는 구조로 추가한다.

## 3. 현재 기준선

현재 구현에는 다음이 포함되어 있다.

- AI 위탁 설정, 거래 로그, 기본 위험 프리셋
- 사용자별 코인원 API 키 조회·복호화 및 코인원 중심 주문 실행
- 최소 신호 신뢰도, 배정 자금 한도, 일간 실현손익 기준 MDD, 동일 종목 중복 매수 차단
- 위탁 거래 로그로 계산한 보유 수량 범위 내 익절·손절 매도
- 토스 OAuth 토큰 요청의 DNS/통신 실패를 워커 전체 종료가 아닌 경고와 해당 주기 건너뛰기로 처리

현재 한계도 명확하다.

- 거래소별 주문 응답과 주문 상태가 같은 형태로 정규화되지 않았다.
- 거래소 실제 잔고·미체결 주문과 내부 거래 로그의 정기 대사가 없다.
- 부분 체결, 주문 응답 유실, 워커 재시작 뒤의 주문 복구가 보장되지 않는다.
- 토스·바이낸스는 AI 위탁 주문 경로가 실주문 어댑터로 완결되지 않았다.

## 4. 아키텍처

### 4.1 책임 분리

| 구성요소 | 책임 |
| --- | --- |
| `ExchangeAdapter` | 인증, 시세, 잔고, 주문, 주문 조회·취소를 거래소 API에 맞게 수행 |
| `ExchangeCapability` | 거래소별 현물/매수/매도/취소/주문조회 지원 여부와 주문 제약 제공 |
| `AiFundRiskService` | 신호·배정 자금·포지션·손실 한도 검증 후 주문 의도 승인 또는 거절 |
| `AiFundOrderService` | 멱등 주문 생성, 상태 전이, 거래소 주문 제출, 체결 반영 |
| `AiFundLedgerService` | 위탁 자금·포지션·실현손익을 주문/체결 사실로부터 계산 |
| `AiFundReconciliationWorker` | 거래소 잔고·미체결 주문을 조회해 내부 원장과 불일치 조정 |
| `AiFundScheduler` | 활성 설정을 순회하며 신호 평가와 복구 작업을 실행 |

어댑터는 전략이나 위험 정책을 알지 못한다. 위험 서비스는 특정 거래소의 HTTP 형식에 의존하지 않는다. 원장은 거래소 잔고 전체가 아니라 AI 위탁운용의 귀속 수량과 현금 흐름만 관리한다.

### 4.2 공통 어댑터 계약

모든 거래소 어댑터는 아래 의미의 메서드를 제공한다.

```python
get_capabilities() -> ExchangeCapability
get_ticker(symbol: str) -> Ticker
get_balance(asset: str) -> Balance
get_open_orders(symbol: str | None = None) -> list[ExchangeOrder]
get_order(order_id: str) -> ExchangeOrder | None
place_market_order(request: OrderRequest) -> ExchangeOrder
cancel_order(order_id: str) -> ExchangeOrder
```

`ExchangeOrder`는 거래소 원본 응답과 별개로 다음 공통 필드를 갖는다: `exchange_order_id`, `client_order_id`, `symbol`, `side`, `requested_qty`, `filled_qty`, `average_fill_price`, `status`, `fee`, `updated_at`.

거래소 고유 응답은 감사와 장애 분석을 위해 별도 원본 필드에 보존하지만, 도메인 로직은 공통 필드만 사용한다.

### 4.3 Capability 정책

`ExchangeCapability`에는 최소 주문 금액, 수량 단위, 가격 단위, 지원 주문 방식, 주문 취소·조회 가능 여부, 현물 거래 지원 여부를 담는다. 주문 요청은 제출 전에 capability 검증을 통과해야 한다.

지원하지 않는 기능은 추정으로 대체하지 않는다. 예를 들어 시장가 매수가 지원되지 않으면 주문 생성은 `REJECTED`로 끝나고 거래소 호출을 하지 않는다.

## 5. 데이터 모델

기존 `admin_ai_fund_configs`와 `admin_ai_trade_logs`는 호환성을 유지한다. 아래 모델을 추가해 실행 사실과 집계 결과를 분리한다.

### 5.1 `ai_fund_orders`

- 설정별·전략별 주문 의도와 거래소 주문 식별자를 보관한다.
- `client_order_id`는 설정 ID, 신호 ID, 주문 유형을 조합한 멱등 키이며 유일해야 한다.
- 상태: `PENDING_SUBMIT`, `SUBMITTED`, `PARTIALLY_FILLED`, `FILLED`, `CANCEL_REQUESTED`, `CANCELED`, `REJECTED`, `FAILED`, `NEEDS_REVIEW`.
- 요청 수량·금액, 체결 수량·평균가·수수료, 거절/오류 사유, 마지막 동기화 시간을 보관한다.

### 5.2 `ai_fund_fills`

- 체결 단위의 불변 기록이다.
- 거래소 체결 ID가 있으면 `(exchange_type, exchange_fill_id)`를 유일하게 보장한다.
- 주문 응답에 개별 체결 정보가 없으면 주문 상태 조회 결과의 누적 체결분만 한 번 반영한다.

### 5.3 `ai_fund_positions`

- AI 위탁운용이 소유한 종목별 수량, 평균 진입가, 누적 실현손익, 마지막 대사 시각을 보관한다.
- 매도 가능 수량은 이 테이블의 수량과 미체결 매도 예약 수량의 차이로 계산한다.
- 거래소 전체 잔고가 충분해도 위탁 포지션 수량을 초과한 매도는 허용하지 않는다.

### 5.4 `ai_fund_reconciliation_runs`

- 대사 시작/종료 시각, 대상 설정, 결과, 불일치 건수, 중단 사유를 보관한다.
- 자동 해결 불가능한 불일치는 `NEEDS_REVIEW` 주문 및 운영 이벤트를 생성한다.

## 6. 주문 처리와 상태 전이

1. 스케줄러가 신호와 활성 설정을 읽는다.
2. 위험 서비스가 운용 모드, capability, 신호 신선도, 일간 손실 한도, 배정 자금, 종목 비중, 중복 주문을 검증한다.
3. 통과 시 멱등 키로 `PENDING_SUBMIT` 주문을 생성한다. 같은 키가 이미 있으면 기존 주문을 반환하고 재주문하지 않는다.
4. LIVE 모드만 어댑터로 주문을 제출한다. PAPER는 모의 체결을 기록하고 CANARY는 설정된 소액 상한 안에서만 LIVE 제출한다.
5. 제출 성공 시 `SUBMITTED`로 전이하고 거래소 주문 ID를 저장한다.
6. 주문 조회 또는 체결 이벤트로 `PARTIALLY_FILLED`와 `FILLED`를 반영하며, 각 신규 체결만 원장에 적용한다.
7. 제출 결과가 유실되거나 상태를 확정할 수 없으면 재주문하지 않고 `NEEDS_REVIEW`로 전이한 뒤 대사 작업에 넘긴다.

상태 전이는 단방향 확정 규칙을 갖는다. `FILLED`, `CANCELED`, `REJECTED`는 종료 상태이며 재시도 대상이 아니다. 네트워크 예외는 주문의 실패를 단정하지 않으며, 우선 거래소 주문 조회로 사실을 확인한다.

## 7. 리스크와 운영 안정성

### 7.1 주문 전 검증

- 활성화된 설정과 유효한 사용자 API 키가 있어야 한다.
- 주문 금액은 배정 자금, 미사용 현금, 거래소 최소 주문 금액, 종목별 최대 비중을 모두 만족해야 한다.
- 신규 매수는 같은 심볼의 미종료 매수 주문 또는 열린 AI 위탁 포지션이 있을 때 차단한다.
- 매도는 AI 위탁 원장 수량에서 미체결 매도 예약 수량을 뺀 범위를 넘지 않는다.
- 신호 생성 시각이 설정된 허용 지연을 초과하면 신규 진입을 하지 않는다.

### 7.2 중단과 복구

- 일간 MDD 한도 초과, 신호 데이터 노후화, 거래소 인증 실패, 대사 불일치 임계치 초과 시 해당 설정은 신규 진입을 즉시 중단한다.
- 긴급 중단은 모든 설정의 신규 제출을 막고, 취소 지원 거래소에는 미체결 주문 취소를 요청한다.
- 주문 취소 요청 자체가 실패하면 주문은 `CANCEL_REQUESTED`에 남기고 대사 작업이 재확인한다.
- 워커는 각 설정·거래소 실패를 격리한다. 한 거래소의 DNS·OAuth·API 장애가 다른 설정의 실행 루프를 종료시키면 안 된다.

### 7.3 대사 기준

활성 설정은 매 주기 또는 최대 60초마다 미체결 주문을 조회한다. 체결 주문은 매 주기 반영한다. 잔고 대사는 최소 5분마다 수행하며, 시스템 재시작 직후에는 주문 제출 전 반드시 수행한다.

불일치는 원본 거래소 데이터와 내부 원장 차이로 계산한다. 자동 조정은 거래소 주문 ID와 체결 사실이 분명한 경우에만 허용한다. 그 밖의 경우 자동 매매를 중단하고 검토 대기로 남긴다.

## 8. 도입 순서와 검증 기준

### 단계 A: 공통 계약과 코인원 이전

- 코인원 구현을 `ExchangeAdapter` 계약으로 옮긴다.
- capability 검증과 공통 주문 모델을 적용한다.
- 기존 AI 위탁운용 단위 테스트가 동일하게 통과해야 한다.

### 단계 B: 주문·체결·포지션 원장

- 주문/체결/포지션 테이블과 멱등성 제약을 추가한다.
- 중복 실행, 부분 체결, 매도 가능 수량 계산을 테스트한다.

### 단계 C: 대사와 운용 모드

- 미체결 주문 동기화, 재시작 복구, 잔고 대사를 구현한다.
- PAPER/CANARY/LIVE별 거래소 호출 여부와 주문 한도를 테스트한다.

### 단계 D: 거래소 확대

- 토스증권과 바이낸스는 capability 지원 범위를 먼저 선언한 뒤, 지원 메서드만 어댑터로 구현한다.
- API 인증 또는 주문 조회가 불안정하면 LIVE 활성화를 차단하고 PAPER 모드만 허용한다.

필수 검증은 단위 테스트, 어댑터 계약 테스트, API 모의 응답 기반 통합 테스트, 데이터베이스 제약 검증, `git diff --check`다. 실제 주문은 별도 승인된 소액 CANARY 환경에서만 검증한다.

## 9. 완료 정의

1차 완료는 코인원에서 모든 주문이 공통 상태 모델과 원장을 통과하고, 재실행해도 중복 주문이 발생하지 않으며, 부분 체결과 통신 실패가 원장 수량을 깨뜨리지 않는 상태다. 토스증권과 바이낸스는 같은 계약을 구현하되, 해당 거래소의 주문·조회·취소 기능이 검증되기 전까지 LIVE를 활성화하지 않는다.

## 10. 구현 상태 (2026-07-22)

### 구현·테스트 완료

- `ai_fund_exchange.py`에 공통 `ExchangeCapability`, `OrderRequest`, `ExchangeOrder`와 거래소 주문 응답 정규화를 추가했다.
- 코인원·바이낸스·토스 클라이언트의 주문 지원 범위를 capability로 조회할 수 있다. 코인원은 LIMIT 전용, 바이낸스는 MARKET/LIMIT, 토스는 기존 주문 조회·취소 계약을 사용한다.
- 스케줄러가 코인원·바이낸스·토스의 사용자별 암호화 API 키를 읽어 각 클라이언트를 생성한다. 토스는 계좌 식별자도 함께 사용한다.
- `ai_fund_orders`, `ai_fund_fills`, `ai_fund_positions`, `ai_fund_reconciliation_runs`용 마이그레이션 파일과 체결·포지션 원장 서비스를 추가했다.
- 주문 제출 전 `PENDING_SUBMIT`을 기록하고, 통신 실패는 재주문하지 않고 `NEEDS_REVIEW`로 전이한다.
- 스케줄러의 ML 신호 식별자를 주문 경로로 전달한다. 같은 설정·거래소·심볼·방향·신호 식별자 조합은 같은 `client_order_id`를 사용하며, 기존 주문이 있으면 거래소에 다시 제출하지 않는다.
- 즉시 부분 체결과 대사에서 확인된 부분 체결은 새 수량만 AI 위탁 포지션 원장에 반영한다.
- 체결 원장 insert가 확인되지 않으면 포지션을 갱신하지 않아, 체결 중복이나 DB 오류가 수량을 이중 반영하지 않도록 했다.
- 스케줄러는 신규 매수 신호보다 먼저 미종료 주문을 대사한다. 거래소 주문을 찾을 수 없거나 조회에 실패하면 검토 대기로 남긴다.
- PAPER 모드는 모의 체결을 기록하고, CANARY 모드는 매수 금액을 설정 상한으로 제한한다.
