# 사용자 친화적 주식·코인 챗봇 응답 UX 설계

## 1. 배경

현재 챗봇의 종목 분석 응답은 시세, 호가, 캔들, 유동성, ML 예측값을 한 번에 나열하는 경향이 있다. 이 방식은 근거를 많이 제공하지만, 일반 사용자가 즉시 이해하기 어렵고 개발자·운영자용 도구처럼 느껴질 수 있다.

주식과 코인은 시장 구조가 다르지만, 사용자가 챗봇에서 기대하는 대화 흐름은 같다. 따라서 공통 응답 UX와 자산별 분석 근거를 분리한다.

## 2. 목표

- 기본 답변은 3~5줄로 제한하고 `결론 -> 쉬운 근거 -> 핵심 위험`순으로 제공한다.
- 초보자도 이해할 수 있지만 경험자에게 필요한 근거를 숨기지 않는 균형형 말투를 사용한다.
- ML은 판단의 중심이 아닌 보조 근거로 취급한다.
- 주식과 코인에 같은 대화 구조를 적용하고, 자산별로 필요한 근거만 다르게 선택한다.
- 일반 채팅 메시지로는 주문 제안을 생성하지 않고, `매매 요청` 폼 제출만 구조화된 주문 진입점으로 인정한다.
- 모든 주문은 제안, 사전 검증, 사용자 승인, 서버 재검증, 실행 절차를 유지한다.

## 3. 비목표

- LLM이 독자적으로 매수·매도를 단정하거나 실거래 주문을 직접 실행하지 않는다.
- 기존 시세·호가·보유자산·ML 수집 로직을 전면 재작성하지 않는다.
- 기본 답변에 모델 버전, 신호 점수, 운영 상태, 원시 API 필드를 나열하지 않는다.
- 이 작업에서 ML 모델 재학습이나 성능 개선을 수행하지 않는다.

## 4. 대화 설계

### 4.1 기본 응답

기본 응답은 다음 세 요소로 구성한다.

1. **결론**: 관망, 분할 접근, 위험 점검 우선 등의 조건부 판단을 한 줄로 제시한다.
2. **쉬운 근거**: 사용자 질문과 관련성이 가장 큰 근거를 최대 두 개만 선택한다.
3. **핵심 위험**: 현재 판단을 바꿀 수 있는 위험 요인 하나와 사용자가 확인할 행동을 알려준다.

답변은 “무조건 매수”처럼 단정하지 않고, “현재 조건에서는 추격 매수보다 분할 접근이 나아 보여요”처럼 근거와 조건을 함께 표현한다.

### 4.2 단계적 정보 공개

기본 답변 아래에 다음 후속 행동을 제공한다.

- `상세 분석`: 가격, 호가, 캔들, 뉴스·공시, ML 보조 신호의 수치와 출처를 확장 표시한다.
- `내 보유 기준`: 보유 수량, 평균단가, 손익률, 포트폴리오 비중을 반영한다.
- `진입 계획`: 분할 기준, 예산, 위험 한도를 정리한다. 주문을 원하면 사용자가 `매매 요청` 버튼을 눌러 폼을 열어야 한다.

사용자가 “확률”, “모델”, “기술적 지표”, “수치로”를 명시하면 첫 답변부터 상세 모드를 사용할 수 있다.

### 4.3 ML 표현 원칙

- 기본 답변: “보조 신호도 상승 쪽이지만 강하지는 않아요”처럼 일반 언어로 요약한다.
- 상세 답변: 상승 확률, 위험 확률, 신호 등급, 모델 버전, 예측 시각을 표시할 수 있다.
- 예측값이 없거나 오래되었을 때는 개발자용 조치 문구를 사용자에게 노출하지 않는다. “현재 활용할 수 있는 보조 신호가 없어 가격과 위험 정보를 중심으로 설명할게요”로 안내한다.
- ML 신호는 결론을 단독으로 결정하지 않고 시세, 유동성, 시장 이벤트, 보유 비중과 함께 해석한다.

## 5. 자산별 근거 선택

### 5.1 주식

질문 의도에 따라 다음 정보 중 최대 두 개를 기본 근거로 선택한다.

- 현재가와 단기 가격 흐름
- 스프레드와 호가 잔량
- 장 운영 상태와 거래 가능 시간
- 최신 뉴스·공시의 핵심 이벤트
- 보유 손익과 포트폴리오 비중
- ML 보조 신호

### 5.2 코인

질문 의도에 따라 다음 정보 중 최대 두 개를 기본 근거로 선택한다.

- 24시간 가격 흐름과 단기 변동성
- 스프레드, 호가 잔량, 예상 슬리피지
- 코인원과 바이낸스 가격 차이와 김치프리미엄
- 거래소 및 선물 포지션 위험
- 보유 손익과 포트폴리오 비중
- ML 보조 신호

## 6. 구현 구조

응답 생성을 다음 세 계층으로 분리한다.

1. **자산 문맥 수집기**: 기존 도구가 시세, 호가, 캔들, 보유자산, 뉴스·공시, ML 데이터를 정규화한다.
2. **공통 응답 정책**: 질문 의도와 세부 모드에 따라 근거 우선순위, 길이, 말투, 위험 표현을 결정한다.
3. **표현 계층**: 기본 3~5줄 답변과 후속 행동을 만들고, 상세 모드에서만 수치와 출처를 확장한다.

공통 응답 정책은 주식과 코인에 동일하게 적용하고, 자산 문맥 수집기만 자산별로 교체할 수 있게 구성한다. 응답 문구를 개별 도구 함수에 중복하여 하드코딩하지 않는다.

## 7. 예외·부분 실패 처리

- 일부 데이터 조회가 실패해도 확인된 데이터로 답변을 계속한다.
- 사용자에게 원문 예외, stack trace, 거래소 raw payload를 노출하지 않는다.
- 핵심 시세를 확인하지 못한 경우 매매 방향을 제시하지 않고 재조회 행동을 안내한다.
- 보유 정보가 없는 것과 API 인증·조회 실패를 구분한다.
- 상세 데이터가 없어도 “모델 스케줄러 확인” 같은 운영자용 조치를 일반 사용자에게 요구하지 않는다.

## 8. 주문 안전

- 분석 답변은 정보 제공이며 주문 실행 명령으로 취급하지 않는다.
- 일반 채팅에서 “삼성전자 10주 사줘”처럼 입력해도 주문 제안, 조건감시 규칙, 실거래 주문을 생성하지 않는다.
- 자연어 주문 의도를 감지하면 챗봇은 주문이 `매매 요청` 폼에서만 가능하다고 안내하고, `매매 요청 열기` 행동을 제공한다.
- 자연어에서 인식한 종목, 매수·매도, 수량, 지정가, REAL/MOCK은 폼의 **임시 입력값**으로만 전달할 수 있다. 인식하지 못한 필드는 안전한 기본값으로 임의 추론하지 않고 빈 값으로 둔다.
- 폼은 자연어로 채워진 필드를 “임시 입력값”으로 표시하고, 사용자가 거래소, 환경, 종목, 매매 구분, 수량, 주문 유형, 가격을 직접 확인하고 제출하게 한다.
- 백엔드는 `structured_order.is_structured_order=true`인 폼 제출만 주문 제안 생성 경로로 허용한다. 같은 문장이 일반 `message`로 전달되면 항상 폼 안내로 처리한다.
- 예전 대화에서 남은 주문 세부정보 확인 대기 상태는 일반 채팅으로 계속 실행하지 않고 폼 안내로 전환한다.
- `진입 계획`은 분석과 계획 수립까지만 수행하며, 제안 생성은 반드시 폼 제출 후에 시작한다.
- 폼 제출 후 제안 생성 전에 가능 수량, 수수료, 슬리피지, 종목 유의사항을 검증한다.
- 실행은 사용자의 명시적 승인 후 서버가 수량, 가격, 계좌, 주문 한도를 다시 검증한 뒤에만 진행한다.
- REAL 환경에는 기존 실거래 하드캡을 적용하고 MOCK 환경은 정책에 따라 분기한다.

## 9. 테스트 기준

- 주식·코인 기본 분석 답변이 3~5줄의 핵심 요약을 제공하는지 검증한다.
- 기본 답변에 모델 버전, 신호 점수, 운영자용 오류 조치가 노출되지 않는지 검증한다.
- 상세 분석 요청에서 숫자 근거와 출처가 누락되지 않는지 검증한다.
- ML 신호가 없는 경우에도 시세·위험 정보로 정상 답변하는지 검증한다.
- 보유 정보 요청 시 REAL/MOCK과 거래소 구분이 유지되는지 검증한다.
- 사용자 승인 없이 주문 도구가 호출되지 않는지 안전 회귀 테스트를 유지한다.
- 자연어 주문 문장이 `create_trade_proposal_from_message()`를 호출하지 않고 폼 열기 행동만 반환하는지 검증한다.
- `structured_order.is_structured_order=true`를 포함한 폼 제출만 기존 사전검증과 제안 생성 경로로 진입하는지 검증한다.
- 임시 입력값이 폼에 표시되더라도 사용자의 폼 제출 전에는 네트워크 주문 요청이 발생하지 않는지 프론트엔드 테스트로 검증한다.
- 주식 질문 “삼성전자 지금 살까?”와 코인 질문 “비트코인 지금 살까?”에 공통 대화 구조가 적용되는지 통합 검증한다.

## 10. 완료 기준

- 주식과 코인 분석이 공통 응답 정책을 사용한다.
- 기본 답변이 3~5줄의 균형형 설명으로 표시된다.
- 사용자가 원할 때만 상세 수치와 ML 정보를 확인할 수 있다.
- 자산별 주요 위험과 보유 문맥이 핵심 답변에 반영된다.
- 일반 채팅의 자연어 주문 요청으로는 주문 제안이 생성되지 않고, `매매 요청` 폼 제출만 주문 진입점이 된다.
- 기존 Human-in-the-Loop 주문 절차와 REAL 환경 안전 가드가 유지된다.
