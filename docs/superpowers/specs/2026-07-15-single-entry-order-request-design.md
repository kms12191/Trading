# 상단 단일 진입 매매 요청 설계

## 1. 목적

챗봇의 자연어 해석을 주문 입력으로 사용하지 않고, 상단 `매매 요청` 버튼에서만 구조화된 매매 제안을 생성한다. 주식, 코인 현물, 바이낸스 USD-M 선물을 같은 3단계 흐름으로 제공하되, 자산과 거래 목적에 맞는 용어·단위·검증 규칙을 적용한다.

본 작업은 데스크톱 환경을 대상으로 하며 모바일 화면 개선은 범위에서 제외한다.

## 2. 핵심 원칙

1. 주문 진입점은 챗봇 상단 `매매 요청` 버튼 하나로 제한한다.
2. 자연어 주문 요청은 추출·임시 저장·폼 자동 입력을 하지 않는다.
3. 사용자가 자유로 입력한 종목 문자열을 주문 심볼로 사용하지 않고, 거래 가능 종목 검색 결과에서 선택하게 한다.
4. 매도와 선물 청산은 서버가 조회한 보유 자산·포지션에서만 선택한다.
5. 사전검증은 제안을 생성하지 않으며, 제안 승인 시 서버가 최신 시세·잔고·포지션·권한을 다시 검증한다.
6. 구조화 요청의 누락값을 `TOSS`, `REAL`, `BUY`, `LIMIT`로 임의 보정하지 않는다.

## 3. 챗봇 진입 정책

- 대화 메시지의 `open_order_form` 행동과 `매매 요청 열기` 버튼을 제거한다.
- 상단 `매매 요청` 버튼은 매번 빈 주문 흐름을 시작한다.
- 자연어 주문 요청에는 "주문은 상단의 매매 요청에서 직접 입력해 주세요."라는 안내만 반환한다.
- 자연어에서 종목, 수량, 가격, 거래소, 방향을 추출해 주문 상태에 보존하지 않는다.
- 제안이 생성된 뒤에만 기존 승인 카드를 대화 타임라인에 표시한다.

## 4. 3단계 사용자 흐름

### 4.1 1단계: 계좌와 거래 목적

사용자가 연결한 계좌만 카드로 표시한다. 카드에는 브로커, 자산 유형, 실거래·모의투자, 주문 가능 잔액, API 권한 상태를 보여준다.

- 주식: Toss, KIS
- 코인 현물: Coinone, Binance Spot
- 코인 선물: Binance USD-M Futures

계좌 선택 후 거래 목적을 고른다.

- 주식·현물: `BUY`, `SELL`
- 선물: `OPEN_LONG`, `OPEN_SHORT`, `CLOSE_POSITION`

선물 화면에서 `매수`, `매도`, `SELL` 같은 포지션 의미가 모호한 표현을 사용하지 않는다.

### 4.2 2단계: 종목과 주문 조건

매수와 선물 신규 진입은 자동완성 결과에서 종목을 선택한다. 검색어는 자유로 입력할 수 있지만, 선택 결과가 없으면 다음 단계로 이동할 수 없다. 결과에는 종목명, 심볼, 마켓, 현재가, 등락률, 거래 가능 상태를 표시한다.

매도와 선물 청산은 계좌의 보유 종목·포지션 목록을 선택한다.

- 매도: 평균 매수가, 현재가, 수익률, 보유 수량, 매도 가능 수량
- 선물 청산: LONG·SHORT, 진입가, 현재가, 미실현 손익, 청산가, 최대 청산 수량

수량 직접 입력 외에 `25%`, `50%`, `전량` 선택을 제공한다. 지정가는 빈 값으로 시작하고, 사용자가 `현재가 적용`을 누른 경우에만 채운다.

자산별 표시 단위는 다음과 같다.

- 국내·미국 주식: 주, KRW·USD
- 코인 현물: 개, KRW·USDT
- USD-M 선물: 계약 수량, USDT

### 4.3 3단계: 사전검증과 확인

`검증하기`는 제안을 생성하지 않고 주문 가능 여부만 확인한다. 결과는 다음 영역으로 나눈다.

- 주문 요약: 계좌, 종목, 방향, 수량, 가격, 예상 금액
- 주문 가능: 잔액, 매도 가능 수량, 필요 증거금, API 권한
- 위험 확인: 수수료, 현재가 대비 입력가 차이, 레버리지, 예상 청산가, 종목 경고

검증 통과 후에만 `매매 제안 만들기`를 활성화한다. 제안 생성 후에도 주문을 실행하지 않고, 대화 타임라인의 승인 카드에서 명시적 승인을 받는다.

## 5. 바이낸스 선물 정책

### 5.1 레버리지와 마진

- 기본 레버리지는 1배이다.
- 기본 마진 모드는 `ISOLATED`이다.
- 종목 선택 후 바이낸스 `leverageBracket`과 심볼 필터를 조회한다.
- 선택 가능 상한은 `min(바이낸스 종목별 상한, 서비스 상한)`이다.
- 서비스 초기 상한은 10배이며, 환경 설정은 이 값을 낮추는 용도로만 사용한다.
- 숫자 자유 입력 대신 해당 종목에서 허용되는 레버리지 선택지만 표시한다.
- 5배 초과 또는 `CROSSED` 선택 시 강화 경고와 재확인을 요구한다.

실거래 1회 10만 원 하드캡은 증거금이 아니라 레버리지 적용 전 명목 주문금액을 원화로 환산해 적용한다.

### 5.2 포지션 모드 변환

| 사용자 선택 | One-way 주문 | Hedge 주문 |
| --- | --- | --- |
| 신규 롱 | `BUY + BOTH` | `BUY + LONG` |
| 신규 숏 | `SELL + BOTH` | `SELL + SHORT` |
| 롱 청산 | `SELL + BOTH + reduceOnly` | `SELL + LONG` |
| 숏 청산 | `BUY + BOTH + reduceOnly` | `BUY + SHORT` |

One-way 청산은 `reduceOnly=true`를 서버가 강제한다. Hedge 모드는 `LONG`·`SHORT`와 `reduceOnly`를 함께 보내지 않고, 선택한 `positionSide`를 고정한 뒤 청산 수량이 해당 포지션 수량을 초과하지 못하게 검증한다.

### 5.3 실거래 가드

- 실거래 신규 롱·숏·청산을 모두 지원한다.
- API Key의 Futures 권한을 사전검증과 승인 시점에 확인한다.
- `BINANCE_FUTURES_REAL_ENABLED=true`가 아니면 실거래 제안 생성과 실행을 차단하고 설정 방법을 안내한다.
- 승인 시 포지션 모드, 최대 레버리지, 심볼 필터, 잔고, 포지션을 다시 조회한다.

## 6. API 및 데이터 경계

### 6.1 주문 입력 API

- `GET /api/trade/order-entry/accounts`: 연결 계좌, 잔액, 거래 권한, 실거래 잠금 상태
- `GET /api/trade/order-entry/symbols`: 계좌에서 거래 가능한 종목 자동완성
- `GET /api/trade/order-entry/holdings`: 매도 가능 보유 종목과 선물 포지션
- `GET /api/trade/order-entry/context`: 최신 시세, 장 상태, 종목 경고, 심볼 필터, 레버리지 상한

상기 API는 프론트엔드에 비밀 키를 반환하지 않으며, 백엔드가 기존 거래소 클라이언트와 보유자산 서비스를 호출한다.

### 6.2 사전검증과 제안

- `POST /api/trade/precheck`: 제안을 생성하지 않고 주문 가능 여부와 예상 비용을 반환한다.
- 검증 성공 결과에는 서버가 서명한 단기 유효 `precheck_token`을 포함한다.
- 계좌, 종목, 방향, 수량, 가격, 레버리지, 마진 모드가 바뀌면 기존 토큰을 무효화한다.
- 구조화 챗봇 요청은 유효한 토큰과 주문 해시가 일치할 때만 `trade_proposals` 제안을 생성한다.
- 승인 API는 실제 주문 전 동일한 공통 검증 서비스를 다시 호출한다.
- 제안 생성과 승인에 멱등성 키를 사용해 중복 제안·주문을 차단한다.

## 7. 서버 검증 규칙

1. 계좌, 거래소, 자산 유형, 거래 목적, 종목, 수량, 주문 유형을 필수로 검증한다.
2. 지정가 주문은 0보다 큰 가격을 필수로 검증한다.
3. 선택한 종목이 해당 계좌의 거래 가능 마켓에 존재하는지 다시 확인한다.
4. 주문 수량을 거래소의 최소·최대·단위 규칙에 맞게 검증한다.
5. 매수 가능 금액, 매도 가능 수량, 선물 증거금과 청산 가능 수량을 서버 조회값으로 검증한다.
6. 실거래 명목 주문금액이 원화 환산 10만 원을 초과하면 차단한다.
7. 시세 조회에 실패하거나 유효 시간이 지난 경우 임의 가격으로 진행하지 않는다.
8. 외부 API 원문 예외와 스택 트레이스를 응답에 노출하지 않고 `format_error_payload()`로 표준화한다.

## 8. 오류 및 재시도 UX

- 오류는 `무슨 일이 발생했는지`, `예상 원인`, `다음 행동`을 포함한다.
- 입력 문제는 필드 인접 오류로 표시하고, 주문 차단 사유는 3단계 검증 결과에 표시한다.
- `수량 수정`, `가격 수정`, `계좌 다시 선택` 행동은 해당 단계로 이동시킨다.
- 최신 시세 시각과 재조회 상태를 표시하고, 검증 토큰 만료 시 `다시 검증하기`를 제공한다.
- 거래 권한·실거래 잠금 문제는 사용자가 확인할 설정 경로를 함께 안내한다.

## 9. 조건감시 범위

익절·손절, 조건매수, `AUTO` 실행은 기본 매매 요청 흐름에서 제거한다. 조건감시는 보유 자산·포지션을 기준으로 하는 별도 설정 흐름에서 설계한다. 선물 자동청산은 롱·숏과 One-way·Hedge 모드를 각각 검증하는 후속 범위로 둔다.

## 10. 테스트 및 완료 기준

### 10.1 백엔드

- 누락된 구조화 주문값이 기본값으로 보정되지 않고 차단된다.
- 주식·현물의 매수·매도와 One-way·Hedge의 신규 롱·숏·청산 변환 행렬을 단위 테스트한다.
- 종목 미선택, 잘못된 수량 단위, 포지션 초과 청산을 차단한다.
- 종목별 동적 상한과 서비스 10배 상한을 모두 검증한다.
- 실거래 명목금액 10만 원 초과, 만료된 검증 토큰, 중복 승인을 차단한다.
- 제안 후 잔고·보유량·포지션이 바뀐 경우 승인 재검증이 주문을 차단한다.
- 외부 실주문 없이 MOCK·TESTNET 통합 테스트를 실행한다.

### 10.2 프론트엔드

- 상단 버튼 외의 주문 폼 진입점이 없다.
- 종목 검색어만 입력하고 결과를 선택하지 않으면 진행할 수 없다.
- 매도·선물 청산이 보유 목록에서만 시작된다.
- 입력값 변경 시 기존 사전검증이 무효화된다.
- 자산별 수량·통화 단위와 선물 진입·청산 용어가 정확하다.
- 제안 생성 전에 사전검증 요약과 위험 정보를 확인할 수 있다.
- 데스크톱 프로덕션 빌드와 3단계 화면 흐름을 검증한다.

## 11. 비범위

- 모바일 화면 최적화
- 자연어 주문 프리필
- LLM의 주문 파라미터 생성
- 기본 매매 흐름 내 조건감시·자동매매
- 서비스 레버리지 상한 10배 초과 운영
